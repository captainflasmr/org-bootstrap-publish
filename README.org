#+title: org-bootstrap-publish
#+author: James Dyer
#+options: toc:2

Generate a Bootstrap 5 static site from one or more Org files.  Each top-level heading becomes a post.  The build writes a =public/= folder you can drop on Cloudflare Pages, Netlify, GitHub Pages, plain S3, or any static host.

This README is a tutorial that walks from a five-minute first site through every layer of customisation the package supports.  Reference tables for every variable and post property live at the end.

* What you get

- Index page with responsive Bootstrap 5 post cards.
- One HTML page per post, rendered through =ox-html=.
- Per-tag pages, a tag cloud, and per-section landings.
- Atom feed at =index.xml= (mirrored to =feed.xml=) and per-tag Atom feeds.
- Hyde-style sidebar — left, right, or top — with title, tagline, navigation, search box, and a light/dark/emacs theme toggle.
- Optional highlight.js syntax highlighting (CDN, opt-out).
- Gallery posts that auto-collect images from a static directory.
- Hugo shortcodes (=youtube=, =video=, =figure=) recognised inline or inside =#+begin_export md= blocks, plus a registry for your own.
- Cross-build HTML cache so warm rebuilds skip work that hasn't changed.
- An async build that keeps the editor responsive on big sites.

* Install

Drop the directory on your =load-path=:

#+begin_src elisp
(add-to-list 'load-path "~/source/repos/org-bootstrap-publish")
(require 'org-bootstrap-publish)
#+end_src

Emacs 28 or newer is required (transient is used for the optional menu).  No package managers, no MELPA, no external dependencies.

* Quickstart: a site in five minutes

The smallest possible site is one Org file with one heading and three lines of configuration.

** Step 1 — write a post

Create =~/site/posts.org=:

#+begin_src org
#+title: Hello World
#+author: Your Name

,* My first post                                                       :hello:
:PROPERTIES:
:EXPORT_HUGO_SECTION: notes
:EXPORT_HUGO_LASTMOD: 2026-04-30 09:00
:EXPORT_FILE_NAME: my-first-post
:END:

This is the body of the post.  Write Org as you normally would —
=src= blocks, lists, =[[file:...][images]]=, the lot.
#+end_src

Note the Org-mode property drawer (=:PROPERTIES:= ... =:END:= block):

- =EXPORT_HUGO_SECTION= — URL prefix.  The post will live at =/notes/my-first-post/=.
- =EXPORT_HUGO_LASTMOD= — sortable timestamp.  Posts on the index are ordered newest-first by this value.
- =EXPORT_FILE_NAME= — the slug.  If omitted, it's derived from the heading text.

The trailing =:hello:= is an Org tag — it becomes a tag page at =/tags/hello/=.

** Step 2 — point obp at the file

#+begin_src elisp
(setq org-bootstrap-publish-source-file "~/site/posts.org"
      org-bootstrap-publish-output-dir  "~/site/public"
      org-bootstrap-publish-site-title  "My Site"
      org-bootstrap-publish-site-url    "https://example.com/"
      org-bootstrap-publish-author      "Your Name")
#+end_src

** Step 3 — build

#+begin_src
M-x org-bootstrap-publish
#+end_src

You'll see something like:

#+begin_example
org-bootstrap-publish: parsed 1 posts from 1 source
org-bootstrap-publish: wrote 1 posts and 1 tags to ~/site/public
#+end_example

** Step 4 — preview

#+begin_src sh
cd ~/site/public && python3 -m http.server 8080
#+end_src

Open http://localhost:8080 and you'll see the home page card grid, the post itself at =/notes/my-first-post/=, the tag page at =/tags/hello/=, the archive at =/posts.html=, and the tag cloud at =/tags.html=.

That's the entire happy path.  Everything below is "now make it do more".

* Live preview with rebuild on save

Manually rebuilding gets old fast.  =M-x org-bootstrap-publish-serve= replaces the previous step:

- Runs an asynchronous full build (editor stays responsive; progress streams to =*obp-build*=).
- Starts a =python3= dev server on port =8080= (configurable via =org-bootstrap-publish-serve-port=).
- Opens your browser at the home page.
- Installs an =after-save-hook= on the source file.  Saving triggers an incremental rebuild of just the post you edited (~0.5 s) and refreshes the tab automatically.

Stop with =M-x org-bootstrap-publish-stop= — it kills the server and removes the save hook.

The live-reload script (a tiny inline =setInterval=) only fires on =localhost=, so it's a no-op on the deployed site.

* Building up: tags, sections, thumbnails

A real site has many posts.  Add more =* Heading= entries to =posts.org= and they'll all be picked up.

** Tags

Tags are just Org tags on the heading line:

#+begin_src org
,* Some thoughts on database indexing                       :postgres:db:
#+end_src

Each tag generates a =/tags/<tag>/index.html= listing every post carrying it, plus a =/tags/<tag>/index.xml= Atom feed.  =/tags.html= shows them all as a tag cloud sized by post count.

The =noexport= tag (configurable via =org-bootstrap-publish-exclude-tags=) skips a heading entirely — handy for drafts.

** Sections

=EXPORT_HUGO_SECTION= sets the URL prefix.  All posts sharing a section show up at =/<section>/= as a paginated landing page (cards by default).  Nested sections (=blog/2026=) work the same way.

** Thumbnails (card images)

Add a thumbnail by setting =EXPORT_HUGO_CUSTOM_FRONT_MATTER+= in the property drawer:

#+begin_src org
:EXPORT_HUGO_CUSTOM_FRONT_MATTER+: :thumbnail /static/notes/my-first-post.jpg
#+end_src

The image must live under one of =org-bootstrap-publish-static-dirs= (see the "Multiple sites" section) so it gets copied into the build output.  The home page card and the section landing card both use it.

** Multiple source files

Set =org-bootstrap-publish-source-files= (plural) instead of =-source-file= and pass a list:

#+begin_src elisp
(setq org-bootstrap-publish-source-files
      '("~/publish/blog.org"
        "~/publish/art.org"
        "~/publish/linux.org"))
#+end_src

Headings from every file are merged, sorted by =EXPORT_HUGO_LASTMOD=, and treated as one stream.  Useful for an umbrella site that aggregates several per-topic blogs.  The save-hook walks the full list, so editing any of them still rebuilds incrementally.

When both =-source-file= and =-source-files= are set, the plural one wins.

* Customising the look

** Layout — left/right sidebar or topbar

Three values for =org-bootstrap-publish-layout=:

| =sidebar=  | (default) vertical bar on the left |
| =rightbar= | same bar mirrored to the right     |
| =topbar=   | horizontal nav above the content   |

#+begin_src elisp
(setq org-bootstrap-publish-layout 'topbar)
#+end_src

The wrapper carries =site-layout-<name>= as a class, so additional layouts can be registered by user CSS without touching the package.

** Three built-in colour themes

Every page ships a sidebar button that cycles through =light= → =dark= → =emacs=.  Choice persists in =localStorage= and is reapplied before first paint, so there's no flash on subsequent loads.

The themes are pure CSS — defined in =assets/style.css= as variable overrides on =[data-obp-theme=...]= selectors.  The toggle just flips that attribute on =<html>=.

** Override CSS variables per site

Want a particular site's sidebar to be navy and orange regardless of which theme the visitor picks?  Set =org-bootstrap-publish-theme-overrides=:

#+begin_src elisp
(setq org-bootstrap-publish-theme-overrides
      '(("obp-sidebar-bg"    . "#1f2a40")
        ("obp-sidebar-fg"    . "#e6ecf5")
        ("obp-sidebar-muted" . "#8b9bb4")
        ("obp-accent"        . "#ff8a4c")
        ("obp-accent-rgb"    . "255, 138, 76")))
#+end_src

Each pair becomes a CSS custom-property declaration emitted against =:root= /and/ every =[data-obp-theme=...]= selector, so the override survives the light/dark/emacs toggle.  Property names accept the leading =--= or omit it.

The full list of variables you can override lives in =assets/style.css= — the defaults are documented there as comments.  The most useful are =obp-sidebar-bg=, =obp-sidebar-fg=, =obp-accent=, =obp-body-bg=, =obp-card-bg=, =obp-card-border=, =obp-link=.

** Background image (with blur and opacity)

Set =org-bootstrap-publish-background-image= to a URL or site-relative path:

#+begin_src elisp
(setq org-bootstrap-publish-background-image "/static/banner.jpg")
#+end_src

The image renders through a fixed =body::before= overlay, so it doesn't scroll with the content and never sits in front of the post body.  When set, the helper also gives =.content-inner= a solid =--obp-body-bg= backing with a small radius — readable text over a busy image.

Tame the image with two extra knobs:

#+begin_src elisp
(setq org-bootstrap-publish-background-blur    8     ;; px, nil for none
      org-bootstrap-publish-background-opacity 0.4)  ;; 0.0 – 1.0
#+end_src

Roughly:

| =blur=  | 4–6 subtle, 8–10 clearly defocused, 15+ abstract wash |
| =opacity= | 1.0 full strength, 0.4–0.6 muted, 0.15–0.25 tinted hint |

Both stack — =blur 12 + opacity 0.2= makes a barely-there abstract backdrop.

* Gallery posts

A heading marked =:EXPORT_HUGO_TYPE: gallery= renders a Bootstrap masonry grid of every image found in =static/<section>/= relative to the source file, sorted by filename descending so date-prefixed files come out newest-first.  The heading body still renders /above/ the grid, so an intro paragraph survives.

#+begin_src org
,* Portraits                                          :portrait:gallery:
:PROPERTIES:
:EXPORT_FILE_NAME: index
:EXPORT_HUGO_SECTION: art--gallery/portraits
:EXPORT_HUGO_TYPE: gallery
:EXPORT_HUGO_LASTMOD: 2024-03-05
:EXPORT_HUGO_CUSTOM_FRONT_MATTER+: :thumbnail /art--gallery/gallery--portraits.jpg
:END:

A short paragraph that appears above the grid.

#+hugo: more
#+end_src

Two conventions:

- =:EXPORT_FILE_NAME: index= collapses the URL to =/<section>/= rather than =/<section>/index/=, matching Hugo's content-bundle convention.
- =static/art--gallery/portraits/= must be in =org-bootstrap-publish-static-dirs= (see "Selective static copy") so the images are copied into the build output.

Recognised extensions: =gif=, =webp=, =jpe?g=, =tiff=, =png=, =bmp=.

* Shortcodes

** Built-in: youtube, video, figure

Three Hugo-style shortcodes are recognised whether they sit on their own line or inside a =#+begin_export md= ... =#+end_export= wrapper:

#+begin_src org
,#+begin_export md
{{< youtube dQw4w9WgXcQ >}}
{{< video src="/static/clip.mp4" >}}
{{< figure src="/static/diagram.png" caption="System architecture" >}}
,#+end_export
#+end_src

They're rewritten to a Bootstrap-friendly =<iframe>=, =<video>=, or =<figure>= during the export pass.  Without this, =ox-html= would silently drop the =md= export blocks.

=/foo= paths in =video= and =figure= shortcodes are rewritten to =<site-path>static/foo= so Hugo's flattened-static convention keeps working.

** Custom shortcodes

Register your own via =org-bootstrap-publish-shortcodes=:

#+begin_src elisp
(defun my/obp-shortcode-tweet (args)
  (let ((id (plist-get args :id)))
    (format "<blockquote class=\"twitter-tweet\"><a href=\"https://twitter.com/x/status/%s\"></a></blockquote>" id)))

(setq org-bootstrap-publish-shortcodes
      '((tweet . my/obp-shortcode-tweet)))
#+end_src

Then in your Org file:

#+begin_src org
{{< tweet id="1234567890" >}}
#+end_src

The handler receives a plist of parsed =key="value"= pairs (under keyword keys, e.g. =:id "1234"=) or =nil= for argless shortcodes.  Return raw HTML; it gets wrapped in a =#+begin_export html= block before =ox-html= runs, so it survives untouched into the final page.

Caveat: handlers must be =fboundp= in whichever process renders the build.  The synchronous build picks them up from your init/script automatically.  The async batch path /does not/ load your init, so any handler not defined inside the package itself will be skipped during async builds.  Use the synchronous =M-x org-bootstrap-publish= when developing custom shortcodes.

* Multiple sites from one Emacs

The recommended pattern is an alist of profiles plus a small selector, kept in a separate file so both interactive Emacs and a batch build script (=bin/web=, GitHub Actions, etc.) can load it.

** The profile alist

Create =~/publish/obp-sites.el=:

#+begin_src elisp
(defvar my/obp-sites
  '((blog
     (org-bootstrap-publish-source-files . ("~/publish/blog.org"))
     (org-bootstrap-publish-output-dir   . "~/publish/blog/public")
     (org-bootstrap-publish-site-title   . "My Blog")
     (org-bootstrap-publish-site-url     . "https://blog.example.com/"))
    (art
     (org-bootstrap-publish-source-files . ("~/publish/art.org"))
     (org-bootstrap-publish-output-dir   . "~/publish/art/public")
     (org-bootstrap-publish-site-title   . "Art")
     (org-bootstrap-publish-site-url     . "https://art.example.com/")
     (org-bootstrap-publish-layout       . topbar))))

(defvar my/obp-managed-vars
  '(org-bootstrap-publish-source-files
    org-bootstrap-publish-output-dir
    org-bootstrap-publish-site-title
    org-bootstrap-publish-site-tagline
    org-bootstrap-publish-site-url
    org-bootstrap-publish-layout
    org-bootstrap-publish-theme-overrides
    org-bootstrap-publish-background-image
    org-bootstrap-publish-background-blur
    org-bootstrap-publish-background-opacity
    org-bootstrap-publish-menu-links))

(defun my/obp-use-site (name)
  "Reset every managed var to its defcustom default, then apply NAME's profile."
  (interactive
   (list (intern (completing-read "Site: "
                                  (mapcar #'car my/obp-sites) nil t))))
  (let ((profile (cdr (assq name my/obp-sites))))
    (unless profile (user-error "No such site: %s" name))
    (dolist (sym my/obp-managed-vars)
      (when-let ((std (get sym 'standard-value)))
        (set sym (eval (car std)))))
    (dolist (pair profile)
      (set (car pair) (cdr pair)))
    (message "obp: switched to %s" name)))
#+end_src

The selector resets every managed var to its =defcustom= default before applying the profile, so each profile only needs to list values that differ from the package defaults — no leakage between sites.

** Per-site serve commands

Generate one =M-x= command per site so you can flip between them without manually stopping/starting:

#+begin_src elisp
(defun my/obp-serve-site (name)
  "Stop any running obp server, switch to NAME, then serve it."
  (interactive
   (list (intern (completing-read "Serve site: "
                                  (mapcar #'car my/obp-sites) nil t))))
  (when (fboundp 'org-bootstrap-publish-stop)
    (org-bootstrap-publish-stop))
  (my/obp-use-site name)
  (org-bootstrap-publish-serve))

(dolist (entry my/obp-sites)
  (let ((name (car entry)))
    (defalias (intern (format "my/obp-serve-%s" name))
      (lambda () (interactive) (my/obp-serve-site name)))))
#+end_src

You'll get =M-x my/obp-serve-blog=, =M-x my/obp-serve-art= etc., automatically — adding a new entry to =my/obp-sites= grows the command list with no further edits.

If you have transient (built into Emacs 28+):

(transient-define-prefix my/obp-menu ()

(global-set-key (kbd "C-c w") #'my/obp-menu)
#+end_src

** Selective static copy

=org-bootstrap-publish-static-dirs= controls which subtrees of the source file's directory are copied into =public/=.  When several sites share one Org tree but only need part of =static/=, list nested paths:

#+begin_src elisp
;; Art site: skip emacs/, kate/, photos/, etc.
(setq org-bootstrap-publish-static-dirs
      '("static/art--gallery" "static/art--videos" "static/art--other"))
#+end_src

Destination directories are created automatically, so deeply-nested entries work without further setup.

** Sidebar links for an umbrella site

An aggregator usually wants the sidebar to point to section landings (=/blog/=, =/photos/=) and specific posts rather than tag pages.  =org-bootstrap-publish-menu-links= renders an alist of =(label . url)= pairs:

#+begin_src elisp
(setq org-bootstrap-publish-menu-links
      '(("gallery"  "/art--gallery/" :style cards)
        ("photos"   "/photos/"       :style cards)
        ("blog"     "/blog/"         :style list)
        ("emacs"    . "/tags/emacs/")
        ("about"    . "/blog/posts--about-me/")))
#+end_src

The long form =(label url :style style)= controls how the destination renders when it points to a generated section landing or tag page (=cards= or =list=).  It's ignored for plain post URLs and external links.

* Section landings

Each distinct =EXPORT_HUGO_SECTION= root automatically gets a =/<root>/index.html= listing every post in that section (and sub-sections), unless a heading with =EXPORT_FILE_NAME: index= already owns the URL.  Pages paginate at =org-bootstrap-publish-posts-per-page= using =/<root>/page/N/= for the second page onwards.

This means a sidebar link to =/blog/= or =/photos/= works without you having to hand-author a landing post for every section.

* Disqus comments

#+begin_src elisp
(setq org-bootstrap-publish-disqus-shortname "your-shortname")
#+end_src

Set per site to keep threads partitioned (e.g. one shortname for the umbrella, another for a sub-site).  The injected embed script skips =localhost= / =127.0.0.1= so the dev server doesn't create stray threads under your account.  =nil= disables.

* Performance: the cross-build cache

Big sites benefit from =org-bootstrap-publish-cache-dir= (defaults to =$XDG_CACHE_HOME/org-bootstrap-publish/=).  Rendered =ox-html= output is keyed by a sha256 of the Org input; warm rebuilds skip =ox-html= entirely for posts that haven't changed.  Stale entries are swept at the end of each build.

Set the var to =nil= to disable caching entirely.  The cache lives outside =output-dir=, so it never gets deployed.

* Async builds

=M-x org-bootstrap-publish-async= forks =emacs --batch -Q= and runs the build there, keeping your editor responsive.  Progress streams to the echo area; the full transcript lands in =*obp-build*=.

=M-x org-bootstrap-publish-async-abort= kills a running child cleanly.

The save-hook rebuild runs synchronously in "fast mode" — it skips per-tag HTML, per-tag feeds, and the site feed, since those embed fully exported post bodies and add ~10 s on a large site.  Fast-mode rebuilds complete in ~0.5 s; the skipped listings refresh on the next full build.

* Deploy

| Cloudflare Pages | =wrangler pages deploy public --project-name=<name>=                        |
| Netlify          | drag-drop =public/=, or =netlify deploy --prod --dir public=                |
| GitHub Pages     | see =workflows/pages.yml= for a CI build, or commit =public/= on =gh-pages= |
| Local push       | =M-x org-bootstrap-publish-publish= rsyncs into =-deploy-dir=, commits, pushes |

For =org-bootstrap-publish-publish= specifically:

#+begin_src elisp
(setq org-bootstrap-publish-deploy-dir    "~/source/repos/site-deploy"
      org-bootstrap-publish-deploy-remote "origin"
      org-bootstrap-publish-deploy-branch "main"
      org-bootstrap-publish-publish-preserve '("CNAME" ".nojekyll"))
#+end_src

A =.nojekyll= is written into the deploy dir so GitHub Pages serves underscore-prefixed paths.

* Reference: post properties

The package reads these from each heading's property drawer:

| Property                           | Use                                                          |
|------------------------------------+--------------------------------------------------------------|
| =EXPORT_FILE_NAME=                 | Slug (lowercased; =index= → URL is =/<section>/=).           |
| =EXPORT_HUGO_SECTION=              | URL prefix.  Posts live at =/<section>/<slug>/=.             |
| =EXPORT_HUGO_LASTMOD=              | Sortable timestamp.  Parses =<YYYY-MM-DD HH:MM>=.            |
| =EXPORT_HUGO_TYPE=                 | =gallery= renders a masonry grid from =static/<section>/=.   |
| =EXPORT_HUGO_CUSTOM_FRONT_MATTER+= | =:thumbnail /path/foo.jpg= → card image.                     |

Heading tags become tag pages.  Headings tagged =noexport= (or whatever's in =-exclude-tags=) are skipped entirely.
